# pseudo-fable-blueprint — 使い方

[English](HOWTOUSE.md) | 日本語

導入後の日常運用ガイド。設計思想と導入手順は [README.ja.md](README.ja.md) を参照。blueprint はプランニングセッション — 仕様を受け取り、要求・設計・チケットに変換する側 — で動く。

## 導入すると何が変わるか

仕様サイズのもの(PRD、バグ修正より大きい機能要望)を渡すと、エージェントは生の仕様から直接設計せず、ゲート付きパイプラインを歩く:

```
INTAKE (spec-interrogate) → DESIGN (design-doc) → PLAN & TICKETS (ticketize) → HANDOFF
```

前フェーズの exit gate が閉じるまで次フェーズには入らない。成果物はチャットではなくファイルに落ちる:

```
docs/plan/<feature-slug>/
├── 01-requirements.md   ← テスト可能な要求台帳、質問、前提、スコープ外
├── 02-design.md         ← recon、代替案、採用設計、故障モード、ADR、プレモーテム
└── 03-tickets.md        ← マイルストーン、依存グラフ、チケット、トレーサビリティ行列
```

## フェーズごとのあなたの役割

| フェーズ | エージェントがやること | あなたがやること |
|---|---|---|
| INTAKE | 仕様の尋問:曖昧さ・矛盾・欠落・暗黙の非機能要件。要求台帳を作成。 | **1 回にまとめられた質問ラウンド**(目安 5 問以下)に答える。各質問には「なぜ重要か」と推奨デフォルトが付くので、「全部デフォルトで OK」も正当な回答。訊かれなかったものは記録付き前提(assumption)になる。 |
| DESIGN | 実コードベースの recon、本質的に異なる設計案 2–3、トレードオフ採点、故障モード分析、インターフェース固定、ADR、プレモーテム。 | 採用案と**却下された代替案**をレビュー(蒸し返し防止はここで効く)。反対はこの段階で — ticketize 後の変更は波及する。 |
| PLAN & TICKETS | walking skeleton 先行のマイルストーン、依存グラフ、テスト可能な受け入れ基準付きチケット、忘れられがちな作業の掃討、トレーサビリティ行列。 | MVP ラインとマイルストーン順を確認。外部トラッカーへの起票(`gh issue create` は外部可視)は承認/拒否 — エージェントは必ず先に確認してくる。 |
| HANDOFF | チケットは pseudo-fable-orchestrate のブリーフに 1:1 で対応(対応表は `ticketize` §6)。または pseudo-fable-lift 下で直接実行。 | 実行者を決める:チケットを委任するか、自分で消化するか。 |

## 典型的な流れ

1. あなた:「課金リニューアルの仕様です:docs/spec-billing.md。計画して」
2. エージェントが仕様全読+リポジトリ recon、R-id 付き `01-requirements.md` を作成(`R1 | p95 < 300 ms at 100 rps | 仕様 §2 | 非機能 | must`)—「速い」のような曖昧語は数値化され、仕様に数値がなければ提案値が入る。
3. 質問ラウンド 1 回。あなたが回答(または「デフォルトで OK」)。
4. ゲートが閉じる → `02-design.md`:代替案 A/B/C を R-id に照らして採点、却下案付き ADR、故障モード、凍結インターフェース。
5. 設計を承認 → `03-tickets.md`:M1 は walking skeleton(最薄の end-to-end スライス)。チケットには Pointers / Acceptance / Non-goals / Depends-on。全 R-id が T-id にトレースされる。
6. ハンドオフ — orchestrate 導入済みなら、チケットはほぼそのままブリーフになる。

## 仕様が途中で変わったら

差分をエージェントに渡すと、**差分に対して** `spec-interrogate` を再実行し、トレーサビリティ行列を通して伝播させる(どの R-id が変わった → どの設計要素 → どのチケット)。変わった仕様からチケットを直接パッチするのは、このパイプラインが防ぎたいアンチパターンそのもの — 見かけたら指摘する。

## 操縦フレーズ

- 「まず spec-interrogate から」(生の仕様から設計し始めたとき)
- 「デフォルトで OK」(質問ラウンドの最速通過)
- 「その決定の却下案を見せて」
- 「R7 はどのチケットに落ちてる?」(トレーサビリティ確認)
- 「仕様が変わった。差分はこれ」(差分再尋問のトリガー)

## 見るべき成果物

- `docs/plan/<slug>/01-requirements.md` — 要求台帳と前提ログ(`A<id> | 前提 | 外れた場合の影響 | 気づく方法`)。前提はあなたのリスク一覧 — 他を飛ばしてもここだけは流し読みを。
- `docs/plan/<slug>/02-design.md` — ADR。ここで固定されたインターフェースが、後で orchestrate の凍結インターフェースになる。
- `docs/plan/<slug>/03-tickets.md` — 末尾のトレーサビリティ行列が「何も落ちていない」ことの証明:T のない R は落ちた要求。

## 想定どおり動かないとき

- **生の仕様から設計する** → 「spec-interrogate が先」。台帳が設計の入力であり、近道はない。
- **質問を 1 つずつ何度も訊いてくる** → トリアージ規則を示す:1 回バッチ・5 問以下・推奨デフォルト付き。残りは記録付き前提にする。
- **要求が曖昧なまま(「堅牢」「使いやすい」)** → pass/fail 形式を要求。間違った提案値は訂正されるが、「速い」は永遠に直らない。
- **チケットにテスト・マイグレーション・ロールアウトがない** → 忘れられがちな作業の掃討を要求 — 各項目は「チケット化/既存チケットに包含/理由付きで除外」のどれかで、沈黙は不可。
