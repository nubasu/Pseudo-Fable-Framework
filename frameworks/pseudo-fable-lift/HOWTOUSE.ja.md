# pseudo-fable-lift — 使い方

[English](HOWTOUSE.md) | 日本語

導入後の日常運用ガイド。設計思想と導入手順は [README.ja.md](README.ja.md) を参照。

## 導入すると何が変わるか

こちらから何も呼び出さなくても、このプロジェクトのエージェントは:

- すべてのタスクを最初のメッセージで**番号付き契約**に言い換え、最後にその契約に対して報告する;
- `finish-gate` を通過するまで「done」と言わない — 完了報告には実行コマンドと実出力が貼られる(「動くはず」ではなく);
- 修正が 1 回失敗したらパッチを積み重ねず、体系的デバッグ(`root-cause-debug`)に切り替える;
- 長いタスクでは状態ファイル(`long-task-state`)を維持し、コンパクションやセッション切れで進捗が消えない;
- 新しいテストは一度 red を見せてから green にする(`test-protocol`)。

常駐コアが常時ステアリングし、5 つのスキルは発火条件を満たしたときだけ読み込まれる。

## あなたの役割

| 場面 | やること |
|---|---|
| タスクを渡すとき | 要求をできるだけ具体的に。エージェントの最初のメッセージが番号付き契約になっているので、**そこで訂正する**(スコープ修正が最も安く済む瞬間)。 |
| 作業中 | 3 ファイル以上・設計判断ありの変更では `deep-plan`(契約/代替案/マイルストーン/プレモーテム)が先に出る。案への反対は diff ができてからではなく、ここで。 |
| 「done」報告 | 契約カバレッジ表とコマンド実出力を確認。証拠がなければ「finish-gate を実行して」と返す。 |
| 長引くバグ | 仮説ジャーナル(`H1: … → ✗/✓`)が出ているはず。仮説なしのリトライをしていたら「root-cause-debug を使って」。 |
| 長時間・複数セッション | `.claude/state/<task-slug>.md` が作られ、更新され続けているか。コンパクション後は行動より先に state ファイルを読み直すはず — していなければ促す。 |

## 典型的なセッション

1. あなた:「レポート画面に CSV エクスポートを追加。空データセットも扱えること」
2. エージェント:番号付き契約(明示要求+後方互換などの暗黙要求)— あなたが訂正または承認。
3. deep-plan の発火条件を超える → 契約、`path:line` 付き recon、代替案 2–3、マイルストーン、プレモーテム。アプローチをここで承認/却下。
4. 小さく可逆なステップで実装。各ステップ後に typecheck / 限定テストで状態確認。
5. 「done」の前に finish-gate — カバレッジ表、新パスの実実行、敵対的 diff 再読、blast-radius の grep。
6. 報告:結論が先頭、契約項目ごとに 変更→検証方法、失敗はそのまま記載。

## 発火条件チートシート

| スキル | 発火条件 | 出てくるもの |
|---|---|---|
| `deep-plan` | 3 ファイル以上/設計判断あり/30 分超の見積り | コーディング前に:契約・代替案比較・マイルストーン・プレモーテム |
| `root-cause-debug` | 修正 1 回失敗/挙動が「変」 | クリーン状態へ戻す、再現、仮説ジャーナル、before/after の証明 |
| `finish-gate` | 完了宣言の直前(必須・例外なし) | Gate A–E。どれか未通過なら状態は「in progress」のまま |
| `long-task-state` | 数時間・複数セッションの作業/コンパクション直後 | `.claude/state/<task-slug>.md` の作成と更新 |
| `test-protocol` | テストの作成・変更 | fail-first の証明、最小モック、境界・エラーパスのカバー |

## 効くフレーズ

発火条件は機械的だが、明示的に呼ぶこともできる:

- 「着手前に deep-plan して」
- 「root-cause-debug でやって」
- 「finish-gate を実行して証拠を見せて」
- 「セッションをまたぐから long-task-state のファイルに計画を書いて」
- 「このテスト信用できないから test-protocol を当てて」

スタックレポート(「棄却済み/残る仮説/次の最良の一手」)は正当な成果物。受け取ったら、盲目的に続けさせるのではなく、質問に答えるか足りないアクセスを与える。

## 見るべき成果物

- `.claude/state/<task-slug>.md` — 長期タスクの状態ファイル:契約、決定、学習事実、失敗したアプローチ、次のアクション。`.claude/state/` は `.gitignore` に追加(ストア直下 README の共通仕上げ参照)。
- 報告そのもの — 証拠テーブルが監査ログになる。中の数値・パスは観測のコピペであり、記憶からの再構成ではない。

## 想定どおり動かないとき

- **証拠なしの「done」** → 「finish-gate」と一言返すだけで大抵直る。繰り返すなら `pseudo-fable-harness` を導入 — Stop フックがゲートを機械的に強制する。
- **スキルが一度も発火しない** → 「使えるスキルを一覧して」で 5 つ見えるか確認。見えなければ配置ミス(`<project-root>/.claude/skills/<name>/SKILL.md` が正)。ストアの `agent-framework-doctor` スキルで診断できる。
- **CLAUDE.md が重くてルールが守られない** → 特に Sonnet では Project specifics をコマンドとエントリポイントだけに絞る(README の「モデル別チューニング」参照)。
- **コンパクション後のドリフト** → 「先に state ファイルを読み直して」。頻発するなら `pseudo-fable-retro` を追加 — session-bootstrap が復帰をプロトコル化する。
