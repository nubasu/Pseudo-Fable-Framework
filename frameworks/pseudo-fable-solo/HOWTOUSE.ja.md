# pseudo-fable-solo — 使い方

[English](HOWTOUSE.md) | 日本語

導入後の日常運用ガイド。設計思想と導入手順は [README.ja.md](README.ja.md) を参照。solo は CLAUDE.md 1 枚だけ — skills ディレクトリがないので、呼び出すものも、発火し損ねるものもない。プロトコルは常駐セクション(§P1–P5)。

## 導入すると何が変わるか

セッションが「自分の後ろに誰もレビュアーがいない」前提の規律あるソロエンジニアとして動く:

- すべてのタスクをまず**番号付き契約**に言い換え、最終報告で各項目を証拠に対応させる;
- 大きめの実装の前には §P1 ディーププラン(契約、recon、代替案 2–3、マイルストーン、プレモーテム);
- 修正が 1 回失敗したら §P2 の仮説ジャーナル付き根本原因デバッグ(パッチの積み重ねはしない);
- **§P3 なしに「done」なし** — 完了ゲート。full diff の「新鮮な目」での敵対的再読を含む(Gate B を先に走らせるのは、書いた直後と読み返しの間に時間を置くため — その距離が 2 人目のレビュアーの代替);
- 長期タスクは §P4 の状態ファイルに、テストは §P5(fail-first・最小モック)に従う;
- さらに Opus 固有の対抗策:選ぶ前に複数候補を保持する、雄弁さより証拠、生成→批判→修正、最後まで全力。

## あなたの役割

| 場面 | やること |
|---|---|
| タスクを渡すとき | 最初の返信が契約への言い換えになる — そこで訂正する。あなたが解消しない曖昧さは、質問ではなく「慣例デフォルト+記録」になる。 |
| 大きな変更 | コードの前に §P1 の出力(代替案込み)が出る。案への異議はこの段階で。 |
| 「done」報告 | カバレッジ表と実出力の貼り付けを確認。なければ「§P3 を実行して」。 |
| しつこいバグ | §P2 ジャーナル(`H1: <原因> — test: <実験> → ✗/✓`)が出ているはず。45–60 分進展がなければスタックレポートで終わるのは正当 — さらに空回りさせるのではなく、その「次の最良の一手」の質問に答える。 |
| 複数セッションの作業 | `.claude/state/<task-slug>.md` があるはず(§P4)。コンパクション後は記憶よりファイルが正 — 記憶で動いていたら読み直させる。 |

## セクション発火チートシート

| 状況 | セクション |
|---|---|
| 3 ファイル以上/設計判断あり/30 分超の見積り | コーディング前に §P1 |
| 修正が失敗した/挙動が「変」 | §P2 |
| コーディングタスクの完了を宣言する直前 | §P3 — 例外なし |
| 数時間・複数セッションのタスク/コンパクション直後 | §P4 |
| テストの作成・変更 | §P5 |

名前で明示的に呼べる:「まず §P1」「§P3 を回してゲートを見せて」「§P4 の状態ファイルを作って」。

## 境界 — solo が不適になるとき

- solo は **Opus 級セッション 1 本が全部自分でやる**構成専用。実装サブエージェントを日常的に起動し始めたら、pseudo-fable-lift + pseudo-fable-orchestrate に移行する(排他ルール:solo と lift は同一 CLAUDE.md に共存しない)。
- 常駐 約 3K トークンは「発火リスクゼロ」の対価。コンテキスト圧が問題になるプロジェクトでは、lift の 2 層構成が軽い代替。

## 見るべき成果物

- `.claude/state/<task-slug>.md` — §P4 状態ファイル(`.claude/state/` は gitignore)。
- 報告 — 契約カバレッジ+証拠。失敗はそのまま、「未確認」は未確認と明記される。

## 想定どおり動かないとき

- **ゲートなしの「done」** → 「§P3」と返す。繰り返すなら `pseudo-fable-harness` を追加 — Stop フックが finish-gate マーカーなしの完了をブロックする(solo の §P3 で条件を満たせる)。
- **最初の案への固着** → §P1 の代替案を明示要求。「複数候補を保持する」は常駐ルールなので、それを引用する。
- **長期タスク終盤の品質低下** → 「最後まで全力」は §How to spend intelligence の 5 番 — 最後の項目に §P3 の再実行を求めると大抵戻る。
- **コンパクション後の混乱** → 「先に §P4 ファイルを読んで。ファイルが記憶に勝つ」。
