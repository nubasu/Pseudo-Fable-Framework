# pseudo-fable-harness — 使い方

[English](HOWTOUSE.md) | 日本語

導入後の日常運用ガイド。設計思想・フックの内部仕様・導入手順は [README.ja.md](README.ja.md) を参照。他のフレームワークと違い、harness はあなたに*間接的に*作用する — エージェントを機械的に拘束し、あなたが体験するのはその副作用。

## 初回チェック(導入直後に一度だけ)

1. **セッションを再起動** — フック登録は起動時に読み込まれる。
2. `/hooks` を実行し、harness の 4 フックが並んでいることを確認(`stop-verify` は設定するまで不活性)。
3. スモークテスト:エージェントに些細なファイル編集をさせ、完了報告なしにターンを終わらせる — stop がちょうど 1 回、ゲート指示付きで弾かれるはず。

## 日常で目にするもの

| 見えるもの | 正体 |
|---|---|
| 完了報告の末尾に `[finish-gate: pass]` | マーカー契約:finish gate(lift の `finish-gate` / solo の §P3)を通過した印。 |
| ターン末尾に理由 1 行+ `[finish-gate: n/a]` | そのターンが完了宣言ではなかった印(ブロック中、入力待ち、非コーディングターン)。 |
| 編集後のターンが終了せず「弾かれる」 | Stop フックの仕事:ファイルを変更したのに最後の編集以降にマーカーがない stop をブロックした。ゲート指示がエージェントに差し戻される。最悪でも 2 回で止まり、無限ループはしない。 |
| サブエージェント返却のたびに受け入れの 1 行ナッジ | PostToolUse フック:統合前に `accept-work` を実行せよというリマインド(読み取り専用の斥候はスルーしてよい旨も書いてある)。 |
| セッション開始時に `.claude/state/` の中身がコンテキストに現れる | SessionStart フックによる状態ファイル注入 — 最新ファイルがインライン化される(60 行/4KB 上限)。**stale 警告**は最新ファイルが 60 分超前のもの、という意味:直近の作業より古い可能性があるので、エージェントは信頼する前に再検証すべき。 |
| チェックコマンドの出力付きで stop がブロックされる | strict verify(有効化した場合):編集後にプロジェクトの実チェックコマンドが失敗した。通るまで完了はブロックされ続ける。 |

## strict verify — 「真実」チェックを入れる

常時稼働の 3 フックが強制するのは*儀式*。strict verify は Gate B(「実際に動く」)の穴を塞ぐ:マーカーは偽装できるが、落ちる typecheck は偽装できない。`.claude/settings.json` の `env` ブロックに、プロジェクトの実チェックコマンドを設定する:

```json
{
  "env": { "PSEUDO_FABLE_HARNESS_VERIFY_CMD": "npm run typecheck && npm test -- --bail" }
}
```

コマンド選定のメモ:

- 編集後にプロジェクトルートから実行される(同梱設定でタイムアウト 300 秒)。毎回の完了時に走らせても苦にならない速さに(typecheck+限定テストが、フル CI より適する)。
- あなたのシェル権限で実行される(`sh -c` / `Invoke-Expression`)— ビルドスクリプトと同じ目でレビューすること。
- 失敗時、エージェントには出力の末尾 1500 文字が渡され、チェックの回避ではなく修正が義務になる。

## キルスイッチ

`PSEUDO_FABLE_HARNESS_DISABLE` は settings.json を触らずに実行時にフックを黙らせる — `stop`, `accept`, `session`, `verify`, `all` のカンマ区切り:

```
PSEUDO_FABLE_HARNESS_DISABLE=accept,verify   # 例:ナッジと strict verify を消音
PSEUDO_FABLE_HARNESS_DISABLE=all             # 全停止(ドキュメントだけのセッションなど)
```

リネーム前の導入(`fable-harness`)は旧名 `FABLE_HARNESS_*` の環境変数を使う — スイッチが効かないように見えたら、導入済みフックスクリプトのヘッダコメントを確認。

## マーカーの正直な読み方

フックが検証するのは儀式であって真実ではない:モデルはゲートを誠実に回さずに `[finish-gate: pass]` を印字*できてしまう* — 偽マーカーの印字は非交渉事項違反と定義されており、その明文の禁止こそが実際の歯止め。だからマーカーは「ゲートを回したという主張」として読み、harness がないときと同様、証拠テーブルの抜き打ち確認は続けること。機械側の答えの半分が strict verify。

## 邪魔になったとき

- **完了宣言でないターンで stop が弾かれる** → そういうターンは理由+ `[finish-gate: n/a]` で終えるのが契約。一度リマインドすればよい — 契約文は CLAUDE.md 追記部にある。
- **シェルコマンド経由の編集(`sed`、`git apply`、生成スクリプト)が検知されない** → 既知の限界:ゲートはファイルツールしか見ない。シェル編集が常態なら、ノイズと引き換えにフックスクリプトのパターンを拡張する — または割り切る。
- **受け入れナッジが煩わしい**(斥候含む全サブエージェントで発火)→ settings から PostToolUse ブロックを削除するか、`PSEUDO_FABLE_HARNESS_DISABLE=accept`。
- **フックが一切発火しない** → セッション再起動 → `/hooks`。Windows でデフォルト(bash)設定を使うには Git Bash が必要 — なければ `settings.hooks.powershell.json` に切り替える。ストアの `agent-framework-doctor` スキルが配線を点検してくれる。
- **フックがエラーを吐く** → 設計上フェイルオープンで、セッションは続行する。繰り返す場合だけスクリプトを調べればよい。

## チューニング

ブロック/ナッジのメッセージ文言はフックスクリプト内にある — CLAUDE.md のルールと同じ流儀で文言を調整し、`.sh`/`.ps1` の双子を揃えて更新し、`.ps1` は ASCII のみを保つ(Windows PowerShell 5.1 は BOM なしスクリプトを ANSI として読むため、非 ASCII は非英語 Windows で文字化けする)。ここにも家訓が適用される:挙動を変えないガードレールは消す。
