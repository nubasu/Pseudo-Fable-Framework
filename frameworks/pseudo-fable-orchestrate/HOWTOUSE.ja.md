# pseudo-fable-orchestrate — 使い方

[English](HOWTOUSE.md) | 日本語

導入後の日常運用ガイド。設計思想と導入手順は [README.ja.md](README.ja.md) を参照。本ガイドはリードセッション(CLAUDE.md に Orchestration セクションを持つ側)を対象とする。ワーカー側の挙動は pseudo-fable-lift(Claude サブエージェント)または AGENTS.md(Codex)が担う。

## 導入すると何が変わるか

リードがソロ実装者ではなくテックリードとして振る舞うようになる:

- **Delegation-first**:実装はデフォルトでワーカー行き。リード自身が手を動かすのは 4 条件(15 分未満の機械作業/実装文脈に深入りしない/誰も自分待ちでない/低リスクで可逆)がすべて成立するときだけ。
- **ブリーフなしに委任なし**:実装サブエージェントを起動する前に、`delegate` スキルが 9 フィールドのブリーフ(Context / Pointers / Contract / Non-goals / Constraints / Gotchas / Done means / Escalation / Report back)を作る。
- **信用でのマージ禁止**:返ってきた成果はすべて `accept-work` を通す — 独立検証と判定(verdict)を経てから統合。
- 並列ファンアウトの前にはインターフェース凍結とファイル分離。統合(マージ後ビルド+全テスト)は常にリード自身の仕事。

## あなたの役割

| 場面 | やること |
|---|---|
| 仕事を渡すとき | タスク分割ではなくゴールを渡す — 分解とルーティングはリードの仕事。プロダクト判断を訊かれたら答える。それ以外はリードが決めて記録する。 |
| ブリーフが出ていくとき | ブリーフは「実際に何が決まったか」の窓。Contract と Non-goals を流し読みする — いま曖昧なブリーフは、後のバウンスになる。 |
| ワーカーが返ってきたとき | 「サブエージェントが完了と言っています」ではなく、証拠付きの accept-work 判定が出るはず。検証なしでマージしそうなら「先に accept-work」。 |
| バウンスが続くとき | 同一タスク 2 バウンスで RECLAIM(ブリーフを書き直して新規ワーカーへ、または引き取り)が強制。3 回目のバウンスをしようとしたら止める — プロトコル違反。 |
| コスト管理 | ファンアウトは実費がかかる。「15 分未満は自分で」ルールとマイクロタスクのバッチ化がレバー。「ワーカーは最大 2 並列で」のように口頭で上限を切ってもよい。 |

## 典型的な委任サイクル

1. あなた:「通知設定機能を実装して。仕様は docs/spec.md」
2. リードが設計判断を自分で行い(判断は委任しない)、インターフェースに沿って分割、共有型を凍結。
3. `delegate` がタスクごとのブリーフを作成。送信前チェック(§3c)で、中のパスとコマンドを実リポジトリに照合。
4. ワーカー実行(Claude サブエージェントはバックグラウンド、Codex は `codex exec`)。
5. 返却ごとに `accept-work`:リードが Done-means コマンドを自分で再実行(または新規の検証エージェントを起動)、full diff を読み、ACCEPT / PATCH / BOUNCE / RECLAIM を判定。
6. 全返却後:リード自身がマージ後ビルド+全テストを実行、台帳を更新、証拠付きで報告。

## 目にする判定(verdict)

| 判定 | 意味 | その後 |
|---|---|---|
| ACCEPT | 全契約項目を独立検証済み | 台帳に記録して統合 |
| PATCH | 本質は正しいが軽微な欠陥あり | リードがその場で修正(深入りするなら 5 行程度のパッチブリーフを同じワーカーへ)。台帳に記載 |
| BOUNCE | 契約未達 — 証拠ベースの具体的フィードバック、同じワーカーへ | 1 タスクにつき最大 2 回 |
| RECLAIM | 2 バウンス後、または根本的な誤解 | ブリーフを書き直して新規ワーカーへ、またはリードが引き取り |

BOUNCE / RECLAIM のたびにブリーフのポストモーテム(「どの一文が欠けていたか」)が走る。これが委任品質を上げるフライホイールで、数サイクル回すとブリーフが目に見えて良くなる。

## 操縦フレーズ

- 「これは委任して。自分で実装しないで」/「このセッションは実装一切禁止で」(ユーザー指示は Delegation-first の判断条項に優先する)
- 「細かい修正はまとめて 1 ブリーフにして」
- 「この diff は別モデル系統のセカンドオピニオンを取って」(クロスモデル敵対レビュー)
- 「台帳を見せて」 — 委任全体の現在地。

## Codex / 外部エージェント

- リポジトリ直下に `AGENTS.md` が必要(同梱の最小版か、pseudo-fable-team の上位互換版)— Codex は CLAUDE.md を読まない。
- Codex へのブリーフは完全自己完結で。非対話実行(`codex exec "<brief>"` — 正確な形式はローカルの `codex --help` で確認)。
- Codex の実行環境はリードと異なりうる:受け入れ時は必ずリード側の環境で再検証。

## 見るべき成果物

- `.claude/state/delegations.md`(pseudo-fable-lift 導入時は long-task-state ファイル内の台帳)— 列:id / executor / brief / status / verdict / evidence。タスク・セッションをまたぐ進捗ダッシュボード。
- ブリーフそのもの — 決定は揮発するチャットではなくここに記録される。

## 想定どおり動かないとき

- **リードが全部自分で実装する** → 「Delegation-first。ブリーフを切って」と促す。逆(些事まで委任)なら 15 分未満ルールを引く。
- **ワーカーの申告のままマージ** → 「accept-work してから統合」。繰り返すなら `pseudo-fable-harness` を追加 — PostToolUse フックがサブエージェント返却のたびに受け入れを促す。
- **ワーカーが同じ失敗を繰り返す** → 疑うべきはワーカーではなくブリーフ。ポストモーテムの「欠けていた一文」とその追記先を確認。プロジェクト固有の罠は常設 Gotchas へ。
- **並列ワーカーが衝突する** → インターフェース未凍結かファイル分離漏れ。次のファンアウト前に `delegate` §4 をやり直させる(重複が不可避なら worktree 分離)。
